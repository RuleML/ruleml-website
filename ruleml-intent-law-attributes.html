Overview

In the design of RuleML we have tried to balance the generality of RuleML's Web rules with their ease of use. Here we describe an approach to improve on both of these for the specification of RuleML 0.88.

Specifically, we strive for the following:

  • Increasing RuleML extensibility via orthogonal, inheritable dimensions:
    • Complementing RuleML's syntax by an orthogonal system of semantic and pragmatic attributes
    • Providing LEGO-like RuleML building-block elements carrying those attributes
    • Allowing OO-like overridable attribute inheritance from parent to child elements
    • Permitting new features, e.g., the 'Turnstile', by attribute combinations (rather than new elements)
  • Omitting role tags where they can be easily inferred from their parent or position (cf. RuleML 0.8 Context, RuleML Lite, and StripeSkipping)
  • Generalizing RuleML to First-Order-Logic RuleML
  • Accomodating requests from SWRL, SCL, and FORUM
  • Re-unifying RuleML Lite with mainstream RuleML

This will be accomplished by establishing a system of dimensions and, in XML, parameterizing RuleML elements by RuleML attributes.

Changes

Renaming of elements:

  • Normal <Rulebase> becomes <And intent="axiom" law="none" closure="universal"> as the root for much inheritance
  • Normal <Imp> becomes <Implies> and, after inheritance, <Implies intent="axiom" law="none" closure="universal">
  • Embedded FOL <Implies> becomes <Implies intent="query" law="any" closure="none">
  • ...

New features by attribute combinations:

  • SWRL RuleML <Imp> becomes, after inheritance, <Implies intent="axiom" law="propositional" closure="universal">
  • <Turnstile><Rulebase>...</Rulebase><Query><body><And>...</And></body></Query></Turnstile> becomes <Implies intent="query" law="any" closure="none"><And intent="axiom" law="none" closure="universal">...</And><And>...</And></Implies
  • ...

The Intent, Law, and Closure Dimensions of Formulas

Dimensions for the 'intent' and 'law' of formulas (e.g., of clauses) are combined for getting a basic formula classification. 'Intent' distinguishes a formula that is asserted as an 'axiom' from a formula that is asked as a 'query', derived as an 'answer', etc. 'Law' distinguishes a formula that can be rewritten ('algebraic' or 'propositional') before its use from a formula that has to be used unchanged ('none').

Specifically:

intent \ law any algebraic propositional none
axiom TBD e.g. for RuleML And e.g. for SWRL RuleML Implies e.g. on RuleML top-level
answer TBD e.g. for RuleML And TBD e.g. in RuleML Implies head
query TBD e.g. for RuleML And e.g. in RuleML Implies body TBD
constraint TBD e.g. for RuleML And e.g. for FOL integrity constraints TBD


A dimension for the 'closure' of formulas is added to the formula classification. 'Closure' distinguishes a formula that is closed ('universal' or 'existential') from a formula that is left open ('none').

The names of these three dimensions (XML attributes) and their values are still somewhat preliminary.

Dimension values need only be specified, via XML attributes, at the uppermost ancestor element for which they hold; processors can use an OO-like attribute inheritance mechanism to carry the attribute values down, via the parent-child chain, from the uppermost ancestor element to all elements that do not mention the same attribute. If an element in this inheritance path does mention the same attribute, its value overrides the value of any ancestor element. This inheritance can be performed statically, say via an XSLT stylesheet, or dynamically, say via a Java application.
